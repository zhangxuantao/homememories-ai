# P0 家庭多用户功能设计

## 背景

当前系统假设单用户在 PC 上操作，但家庭回忆管理的真实场景是多用户协作。本次设计补齐三个 P0 功能，让家庭成员能参与内容共建，而非仅被动浏览。

三个功能的关系：**扫码访问降低第一道门槛 → 手机上传让家人成为内容贡献者 → 批量操作提高整理效率**。

---

## 功能一：手机上传

### 用户故事

老婆用手机在客厅翻看家庭照片，想起上周末拍的宝宝视频还在手机里。点 "+" → 选照片 → 上传，几秒钟后照片出现在首页"最近添加"中。无需碰电脑，全程在手机上完成。

### 入口

- **移动端**：底部导航栏新增 "+" 按钮，居中放置，视觉上做强调（主色填充圆形）
- **PC 端**：顶部工具栏右侧 "上传" 文字按钮，点击弹出 Modal 对话框

### 上传面板

```
┌──────────────────────────────┐
│  上传照片/视频            ✕  │
│                              │
│  ┌──────────────────────┐   │
│  │    点击或拖拽选择      │   │  ← PC端支持拖拽
│  │   JPG/PNG/MP4/MOV   │   │
│  └──────────────────────┘   │
│                              │
│  已选 3 张                   │
│  ┌───┐ ┌───┐ ┌───┐         │
│  │ ✓ │ │ ✓ │ │   │         │  ← 缩略图预览，角标×可移除
│  └───┘ └───┘ └───┘         │
│                              │
│  [████████░░] 67%           │
│  正在上传: IMG_4321.jpg     │
│                              │
│  ✓ 3 张已上传，后台处理中... │
│  [去查看]                    │  ← 跳转到最近添加
└──────────────────────────────┘
```

### 交互细节

- 文件选择器 `accept="image/*,video/*"` + `multiple`，移动端浏览器自动调起系统相册
- 支持摄像头直拍（`capture="environment"`）作为备选入口
- 单次上限 100 张，超出弹 toast："单次最多上传 100 张"
- 上传采用并发 3 张的策略，避免浏览器内存压力
- 每张上传完成后显示 ✓，全部完成后汇总
- 失败的照片单独标注，支持"重试失败项"
- 上传完成后后台异步处理：缩略图生成 → EXIF 解析 → dHash 计算 → AI embedding 生成，不阻塞用户

### 后端接口

```
POST /api/media/upload
Content-Type: multipart/form-data
Body: files[] (多个文件，单文件最大 100MB)

Response 200:
{
  "uploaded": [
    { "id": 123, "filename": "IMG_4321.jpg", "media_type": "image" },
    { "id": 124, "filename": "VID_0010.mp4", "media_type": "video" }
  ],
  "failed": [],
  "processing": true
}
```

上传后自动触发的处理链：保存文件到 media_root → INSERT media 记录 → 异步 celery-style 任务链：生成缩略图 → 计算 dHash → 生成 embedding → 重建 FAISS 索引

### 涉及文件

| 类型 | 文件 | 改动 |
|------|------|------|
| 新建 | `frontend/src/components/UploadPanel.tsx` | 上传面板组件 |
| 修改 | `frontend/src/components/layout/MobileNav.tsx` | 新增 "+" 按钮 |
| 新建 | `backend/app/routers/upload.py` | 上传 API 端点 |
| 新建 | `backend/app/services/upload_service.py` | 文件保存 + 异步处理调度 |
| 修改 | `backend/app/main.py` | 注册上传路由 |

## 功能二：扫码访问

### 用户故事

老公在电脑上部署好系统。老婆想用手机看——不用打字输入 IP，扫一下屏幕上的二维码直接打开。

### 入口

设置页中新增"局域网访问"卡片，放置在"系统信息"上方。

### UI

```
┌──────────────────────────────┐
│  局域网访问                   │
│                              │
│  ┌────────────┐              │
│  │            │              │  ← 二维码（前端 qrcode 库生成）
│  │   QR 码    │              │
│  │            │              │
│  └────────────┘              │
│                              │
│  🌐 http://192.168.1.5:5173 │  ← 当前地址，点击可复制
│      [复制地址]              │
│                              │
│  小字提示：用相机或浏览器扫码，│
│  微信扫码请点击右上角"在浏览器 │
│  中打开"体验更佳。            │
│                              │
│  ✅ 当前已连接局域网          │  ← 状态指示
│  ⚠️ 未检测到局域网，请确认    │
│     WiFi 已连接              │
└──────────────────────────────┘
```

### 交互细节

- 二维码内容 = `window.location.origin`（自动包含当前协议+IP+端口）
- 使用 `qrcode` npm 包（约 10KB）在前端生成 SVG，无需后端
- "复制地址" 按钮点击后 `navigator.clipboard.writeText()` + toast "已复制"
- 状态检测逻辑：hostname 是 192.168./10./172.16-31. 开头 → ✅；是 localhost/127.0.0.1 → ⚠️ "当前通过本机访问，请确认已连接局域网"
- 前端开发模式下 vite proxy 会返回 localhost，需在设置页通过 API 获取真实 IP（新增 `/api/admin/server-info` 端点）

### 后端接口

```
GET /api/admin/server-info

Response:
{
  "hostname": "DESKTOP-ABC",
  "lan_ip": "192.168.1.5",
  "port": 8501,
  "frontend_port": 5173
}
```

### 涉及文件

| 类型 | 文件 | 改动 |
|------|------|------|
| 修改 | `frontend/src/pages/SettingsPage.tsx` | 新增局域网卡片区块 |
| 新增 | `frontend/src/components/ui/QrCode.tsx` | 二维码组件 |
| 新增依赖 | `qrcode` | npm 包 |
| 修改 | `backend/app/routers/admin.py` | 新增 `/server-info` 端点 |

## 功能三：批量操作

### 用户故事

用户在搜索结果中找到 8 张不错的照片，想一起加入"宝宝成长"相册。长按第一张进入选择模式，连续勾选 8 张，底部点"加入相册"，选目标相册，完成。无需一张一张操作。

### 入口

- **移动端**：照片网格中长按任意照片 300ms → 进入选择模式，配合 `navigator.vibrate(15)` 触觉反馈
- **PC 端**：hover 照片时左上角显示勾选框，点击勾选框进入选择模式；也支持右键菜单 "选择多张" 入口

### 选择模式 UI

```
┌──────────────────────────────┐
│  ← 已选 8 项          [全选] │  ← 顶栏替换原页面导航
│                              │
│  ┌───┐ ┌───┐ ┌───┐         │
│  │ ✓ │ │ ✓ │ │   │         │  ← 未选中的半透明 + 灰色遮罩
│  │   │ │   │ │   │         │
│  └───┘ └───┘ └───┘         │
│  ┌───┐ ┌───┐ ┌───┐         │
│  │ ✓ │ │ ✓ │ │ ✓ │         │  ← 选中的正常显示 + 蓝边框 + ✓
│  └───┘ └───┘ └───┘         │
│  ...                         │
│                              │
│  ┌────────┐ ┌────┐ ┌─────┐ │  ← 底部固定操作栏
│  │加入相册 │ │下载 │ │删除  │ │
│  └────────┘ └────┘ └─────┘ │
└──────────────────────────────┘
```

### 交互规则

| 动作 | 效果 |
|------|------|
| 长按照片 | 进入选择模式，该照片被选中 |
| 点击未选中照片 | 选中，顶栏计数 +1，选中态视觉切换 |
| 点击已选中照片 | 取消选中，顶栏计数 -1 |
| 滑动经过照片 | 手指滑过的照片自动选中（常见相册交互模式） |
| 点击"全选" | 选中当前视图所有项，按钮变为"取消全选" |
| 点击"←"或"取消" | 退出选择模式，清除所有选中 |
| 按 ESC / 返回键 | 退出选择模式 |
| 完成任一操作后 | 自动退出选择模式 |

### 底部操作按钮

**加入相册**
点击 → 底部弹出半屏面板，列出已有相册 + "新建相册"入口 → 选择相册 → 调用 API → toast "已加入 8 张到「宝宝成长」"

**下载**
- ≤ 5 张：前端逐张触发浏览器下载
- > 5 张：调后端 `/api/media/export-zip`，生成临时 zip 包，触发浏览器下载

**删除**
弹出二次确认弹窗："确定删除这 8 张照片？此操作不可恢复。建议先确认是否有备份。" 确认后调用 DELETE 接口，完成后 toast "8 张已删除"，被删照片从页面移除。

### 适用范围

| 页面/场景 | 是否支持批量 | 备注 |
|-----------|-------------|------|
| 搜索结果 | ✅ | 核心场景 |
| 时间线事件展开 | ✅ | 事件内照片网格 |
| 人物相册详情 | ✅ | 同一人物所有照片 |
| 模糊照片列表 | ✅ | 批量删除最常用 |
| 重复照片组 | 部分 | 仅支持单组删除，批量需适配 |
| Lightbox 相似照片 | ❌ | 结果量太少，不划算 |
| 首页卡片流 | ❌ | 卡片流不适合选择模式 |

## 分期实施建议

三个功能可独立实施，但建议按以下顺序：

1. **扫码访问** — 改动最小，无后端依赖，可独立交付，为后续手机功能铺路
2. **手机上传** — 依赖扫码访问降低门槛，让家人真正参与
3. **批量操作** — 依赖上传增多后，整理需求自然产生
